# Contributing to KCC Analysis

Thank you for contributing to the KCC Analysis project! This guide will help you get started.

## Development Setup

### 1. Fork and Clone
```bash
git clone <your-fork-url>
cd "KCC Analysis"
```

### 2. Create a Feature Branch
```bash
git checkout -b feature/your-feature-name
```

### 3. Set Up Development Environment

**Python Environment:**
```bash
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate
pip install -r requirements.txt
```

**Dashboard Development:**
```bash
cd dashboards
npm install
npm run dev
```

## Project Workflow

### Working with Data

**Large data files are git-ignored**. To work with the full dataset:
1. Contact the project maintainer for data access
2. Place files in the appropriate directories:
   - `data/processed/` for processed datasets
   - `data/raw/` for raw data

### Running Analysis Scripts

**Before running analysis:**
- Ensure you have the required data files
- Update configuration variables if needed (crop name, parameters)
- Scripts now use relative paths and are portable

**Example:**
```bash
# From project root
python analysis/saturation_analysis.py
```

### Dashboard Development

**Data files for dashboard:**
- Generated by analysis scripts
- Located in `dashboards/src/data/`
- Git-ignored but necessary for local development

**Generate data files:**
```bash
python analysis/saturation_analysis.py  # Generates saturationData.json
# Other scripts generate cropData.json
```

## Code Style

### Python
- Follow PEP 8 guidelines
- Use meaningful variable names
- Add docstrings to functions
- Keep functions focused and modular

### JavaScript/React
- Use ES6+ syntax
- Follow React best practices
- Use meaningful component names
- Keep components small and reusable

## Commit Guidelines

### Commit Message Format
```
<type>: <subject>

<body>

<footer>
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code style changes (formatting, etc.)
- `refactor`: Code refactoring
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples:**
```
feat: add crop comparison feature to dashboard

Implemented side-by-side crop comparison with FAQ tables
and query distribution charts.

Closes #123
```

```
fix: correct saturation curve calculation

Fixed bug where cumulative percentages were not properly
normalized for crops with missing clusters.
```

## Pull Request Process

1. **Update your branch** with the latest main:
   ```bash
   git fetch origin
   git rebase origin/main
   ```

2. **Test your changes thoroughly**:
   - Run analysis scripts if you modified them
   - Test dashboard locally if you modified UI
   - Check for errors and warnings

3. **Create a Pull Request**:
   - Use a clear, descriptive title
   - Describe what changes you made and why
   - Reference any related issues
   - Add screenshots for UI changes

4. **Code Review**:
   - Address review comments promptly
   - Make requested changes in new commits
   - Don't force-push after review has started

5. **Merge**:
   - Squash commits if requested
   - Delete your branch after merge

## Adding New Features

### Adding New Analysis Scripts
1. Place in `analysis/` directory
2. Use relative paths (see existing scripts)
3. Add docstrings and comments
4. Update README.md with usage instructions

### Adding Dashboard Components
1. Place in `dashboards/src/components/`
2. Follow existing component structure
3. Add prop validation
4. Update App.jsx to import and use

### Adding New Visualizations
1. Generate in `outputs/figures/`
2. Use consistent styling (dark theme)
3. Update documentation

## Testing

### Python Scripts
- Test with actual data files
- Verify output files are generated correctly
- Check for edge cases (empty data, missing files, etc.)

### Dashboard
```bash
cd dashboards
npm run lint      # Check for linting errors
npm run build     # Test production build
npm run preview   # Preview production build
```

## Getting Help

- **Questions**: Open a GitHub Discussion
- **Bugs**: Open a GitHub Issue with:
  - Clear description
  - Steps to reproduce
  - Expected vs actual behavior
  - Environment details (OS, Python version, etc.)
- **Feature Requests**: Open a GitHub Issue with:
  - Clear use case
  - Proposed solution
  - Alternative approaches considered

## Code of Conduct

- Be respectful and inclusive
- Provide constructive feedback
- Focus on the code, not the person
- Assume good intent
- Help others learn and grow
